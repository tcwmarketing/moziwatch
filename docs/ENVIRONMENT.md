# Environment configuration

No secret values belong in source control. Copy `.env.example` to `.env.local` for local work and configure the equivalent encrypted variables on the deployment platform.

## Database and Supabase

| Variable                   | Development                                                                | Testing                                                                                                        | Production                                                                                                                                               | Provider and where to obtain it                                                                                                          | Can the app operate without it?                                                                                                            |
| -------------------------- | -------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------ |
| `DATABASE_URL`             | Required to use any database-backed page/API. Local Docker is the default. | The general unit suite does not need it; integration tests replace it with `TEST_DATABASE_URL`.                | Required. Use Supabase Connect → Transaction pooler for serverless/auto-scaling deployment, or Direct/Session mode for a persistent server.              | Supabase dashboard → project → **Connect** → connection string. Insert the database password and retain `sslmode=require` when supplied. | Only static pages and the fail-closed map shell work; reports, auth, campground data, forecasts, and admin do not.                         |
| `DIRECT_DATABASE_URL`      | Optional when local `DATABASE_URL` is already direct.                      | Not used for the dedicated test migration.                                                                     | Recommended and normally required operationally for migrations/backups. Use Supabase Direct connection, or Session pooler from an IPv4-only environment. | Supabase dashboard → **Connect** → Direct connection or Session pooler.                                                                  | Runtime can operate without it; `db:migrate` falls back to `DATABASE_URL`, which is not recommended when that URL is a transaction pooler. |
| `TEST_DATABASE_URL`        | Optional.                                                                  | Required for `npm run test:integration`. Must identify a separate PostGIS-enabled disposable database/project. | Never set to production.                                                                                                                                 | Create a second Supabase project/database dedicated to automated tests, then copy its connection string from **Connect**.                | Yes; only the database concurrency test is skipped/unavailable.                                                                            |
| `TEST_DATABASE_DISPOSABLE` | Leave `false` unless running the dedicated integration test.               | Must be exactly `true` for test migration/execution.                                                           | Must not be enabled in the production environment.                                                                                                       | Operator-set safety acknowledgement, not supplied by a service.                                                                          | Yes, except the destructive-capable integration test refuses to run.                                                                       |

The application uses Better Auth and Drizzle directly against PostgreSQL. It does not use Supabase Auth, Storage, Realtime, Edge Functions, or the Supabase JavaScript client. Therefore `SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_URL`, `SUPABASE_ANON_KEY`, and `SUPABASE_SERVICE_ROLE_KEY` are neither read nor required.

Enable PostGIS in Supabase under Database → Extensions → `postgis`. The migration also runs `CREATE EXTENSION IF NOT EXISTS postgis` and then creates generated geography points and GiST indexes.
